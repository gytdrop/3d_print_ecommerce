   - Total pincodes generated by seed scripts: `110 + 15 = 125` pincodes.
3. **Application initialization safety**:
   - `db.go` successfully embeds the DDL and seed SQL using the `embed` library (line 8, lines 24-28).
   - Initialization (`InitDB`) executes both files inside a single database transaction, ensuring atomicity (all-or-nothing).
4. **Foreign Key Integrity Challenge**:
   - Orders require a reference to `pincodes(pincode)`.
   - If an order is created for a standard pincode outside the 125 seeded zones, the DB insert will throw a foreign key constraint violation.
   - In `backend/internal/handlers/handlers.go`, this is handled at the application layer by dynamically registering standard pincodes upon order placement (lines 194-203):
     ```go
     _, err = db.DB.ExecContext(r.Context(),
         `INSERT INTO pincodes (pincode, tier, is_eligible_express, message)
          VALUES ($1, 'standard', false, 'Standard shipping available.')
          ON CONFLICT (pincode) DO NOTHING`, req.Pincode)
     ```
     This mitigates the risk of foreign key crashes for standard orders.

### Minor Quality Improvement recommendation:
In `backend/internal/db/db.go`, the connection retry loop calls `sql.Open` inside the loop. `sql.Open` should be called once outside the loop, and the loop should only execute `db.Ping()`. Calling `sql.Open` repeatedly leaks unclosed `*sql.DB` handles on intermediate failure.

### Findings:
- **[Minor] Finding 1: DB pool recreation in retry loop**
  - **Where**: `backend/internal/db/db.go:44-57`
  - **Why**: Re-creating the connection pool with `sql.Open` inside a loop can lead to file descriptor and resource leaks if intermediate attempts initialize the pool but fail `db.Ping()`.

## 7. Detailed Adversarial Challenge Report

### Overall risk assessment: LOW

### Challenges:
- **[Medium] Challenge 1: Foreign key constraint violation on standard orders**
  - **Assumption challenged**: All orders will be placed within the 125 pre-seeded pincodes.
  - **Attack scenario**: A user places an order using a standard pincode (e.g. `110001` for Delhi). Because the pincode table only contains 125 seeded codes, inserting the order directly fails the foreign key constraint.
  - **Blast radius**: Prevents any checkout outside the pre-seeded areas.
  - **Mitigation**: Confirmed that the application handles this in `handlers.go` by running an upsert `INSERT INTO pincodes ... ON CONFLICT DO NOTHING` for the incoming pincode prior to inserting the order.

- **[Low] Challenge 2: Alphanumeric or international pincode values**
  - **Assumption challenged**: Pincodes will always be 6 digits and numeric.
  - **Attack scenario**: The system expands to support international shipping (e.g. UK zip codes like `EC1A 1BB`).
  - **Blast radius**: The `pincodes` table has `pincode VARCHAR(6) PRIMARY KEY` and validation functions enforce a length of exactly 6 numeric characters, which blocks international customers.
  - **Mitigation**: Keep validation as-is for the current India-focused phase; plan table schema migration if international expansion is initiated.
